Onboard-to-ENS Subname Manager

A full platform where a web3-novice signs in with email, gets (or connects) an ENS name, and an organization can auto-issue member subnames — with multisig-protected parent control and verifiable, oracle-gated issuance.

ENS · primary Chainlink CRE · verifiable issuance gate Ledger · device-backed approval Privy · wallet infra (no prize)
Updates since this plan — as built (June 2026) Parts of this plan were superseded during the build. Where the sections below conflict, these points win:
  • NameStone dropped. It requires a parent the platform controls plus per-domain enablement — incompatible with self-serve, user-owned names. Subname issuance is now onchain-first on the ENSv2 PermissionedRegistry: each parent issues through its own UserRegistry subregistry, deployed as a proxy via the VerifiableFactory (same pattern as the per-account resolver). The offchain option is deferred to our own CCIP-Read gateway (Durin-style), not a third party.
  • Platform-mediated issuance. A member holds no registrar role, so the platform issues on their behalf — but only after verifying eligibility server-side: Privy access token → verified email → enrolled-org domain match. A platform key holding ROLE_REGISTRAR mints the subname to the member's own wallet. Email = eligibility, never authority.
  • Enrollment is database-backed (Supabase): a domain → parent table plus an index of issued subnames; lookups are server-only. Public email providers (gmail, outlook) route to a self-serve path rather than org enrollment.
  • Names resolve and carry profiles. Subnames are minted pointing at a platform resolver with an address record set, so they resolve to the owner's wallet; members edit text records (display name, avatar, url) from the manager.
  • Golden path, as built: email login → embedded wallet → eligibility check → claim a real onchain subname → it resolves → edit profile. Validated live on Sepolia. Reference implementation for the onchain calls: gskril/ens-cli + ensdomains/namechain.

01Thesis & what's novel

Existing subname tooling (Namespace, NameStone, ensub.org) all begin after the user already owns an ENS name and has a wallet. The novel surface is everything before that line, plus the org auto-issuance flow on top:

You build (nobody else does)
  • Email → wallet → name onboarding for users with zero web3 knowledge
  • Three entry doors unified into one session model
  • Org email auto-issuance: alice@acme.com → alice.acme.eth if the org is enrolled
  • Manager UI on top of ENSv2's per-name registries & role model
Already solved (you consume, don't rebuild)
  • Offchain subname resolution → NameStone (CCIP-Read)
  • Onchain subname primitives → ENSv2 / NameWrapper
  • Embedded wallets & email auth → Privy
  • SIWE message/verify → spruceid/siwe or an auth framework plugin
Prize logic ENS is the anchor (target Most Creative Use of ENS — subnames-as-org-identity is on their own example list — plus the $6k Integrate-ENS pool). Chainlink CRE adds a track via a real on-chain state change (issuance gated on a consensus-verified eligibility signal). Ledger adds a track as the device-backed approval layer for high-risk parent operations. Privy stays in as wallet-creation infrastructure only — we use it, but don't submit for its prize since we touch none of its new features (Earn / deposit addresses / Agent CLI). All three prize tracks reinforce the manager rather than pulling it off-thesis.
Build target & production path Everything is built and demoed on Sepolia testnet, with the architecture deliberately portable to Ethereum mainnet later — same contracts, same flows, only addresses and gas economics change. Worth stating in the pitch: judges value a credible path to production, not just a testnet toy.

02User types & the authority model

The single most important design decision. There are three distinct actors, and the trap is letting email grant parent-name authority. It must not.

ActorHow they're authenticatedWhat they can do
Org Admin
controls the parent		 Wallet control of the parent .eth name, proven via SIWE — not email. Control posture is the owner's choice: solo key (optionally Ledger-backed) or Safe multisig. Enroll the org, set issuance policy (onchain/offchain), grant/revoke manager roles, transfer/revoke the parent, reassign subtrees.
Manager
delegated operator		 Holds an ENSv2 role granted by an admin (a multisig action). Issue / revoke member subnames, edit records. Cannot transfer the parent or remove other admins.
Member
subname holder		 Verified email whose domain matches an enrolled org (Privy gives a verified email + domain on the session). Receive & control them.org.eth, edit their own profile, create their own sub-subnames.
Security hole to avoid "First person from an org to sign in becomes the parent owner" is unsafe: any current/compromised @acme.com address could seize the namespace, and shared/rotating club emails make it a race. First-sign-in is only a bootstrap trigger, never the authority. Authority over the parent always traces to wallet control of the .eth name (existing owner) or to whoever completes registration (new org), and that account should be a Safe.
The clean separation Email = eligibility ("you may receive you.acme.eth").  Wallet / multisig = administration ("you may govern acme.eth"). Keep these two powers in different boxes and the attack surface collapses.

Frontend UX requirement — info icon on every action

The product premise is web3 novices who don't know what any of this means. So contextual help isn't polish — it's core to the value proposition. Requirement: every button and function carries an info icon (ⓘ) with a tooltip/popover explaining what it does and when to click it, in plain language with zero jargon.

Pattern

A reusable <InfoTip> component next to every control: hover/tap reveals one or two sentences. Plain words ("This creates a free web address for a member" — not "mints an offchain CCIP-Read subname"). Optionally a "Learn more" link for the curious.

Examples
  • Onchain/offchain toggle ⓘ — "Offchain = free & instant. Onchain = costs a small fee but is permanent and fully owned by the member."
  • Issue subname ⓘ — "Gives this person their own name under your organization, e.g. alice.yourorg.eth."
  • Multisig toggle ⓘ — "Require more than one person to approve big changes. Safer for real orgs; skip it if you're solo."

Call this out explicitly in the demo narration — "every control explains itself" is a concrete answer to "how is this usable by someone with no web3 knowledge," which is your core differentiator.

04Auth-path decision tree

Two real mechanisms; the third door is a friendly front-end on the second.

Email
(new user)
Privy issues a verified email + provisions an embedded wallet silently. Session = {address, verifiedEmailDomain}. This is your primary funnel and the whole reason the product is novel. → On success, check org-domain match.
Connect
wallet
Standard connect (RainbowKit/ConnectKit). You have an address immediately, possibly already owning names. No email unless they add one. → Skip to "owns a name?".
Type .eth
name
Typing a name does not authenticate — anyone can type vitalik.eth. Flow: resolve name → address, then run SIWE (EIP-4361) so they sign a nonce'd message proving control of that address. Verify signature ⇒ mint a normal session cookie/JWT. The name is just a friendlier lookup over SIWE.
Don't hand-roll SIWE Use spruceid/siwe or an auth-framework SIWE plugin. The library enforces nonce + domain + chain-ID binding and expiry; rolling your own signature recovery without that binding is a replay foot-gun.

05Org enrollment & delegation mechanics

Enrollment is the act that grants your platform permission to issue under a parent name. It also fixes the issuance policy and the control posture (§2). Three cases at registration time:

A · Org already owns acme.eth

Admin connects the controlling wallet, signs SIWE to prove control, sets the control posture (solo or Safe) + initial managers. Control of the name is the proof. Email never touches authority here.

B · Org has no name yet (your novel case)

Platform walks them through registering acme.eth on the v2 ETHRegistrar (0x8c2E…fFcA). The registering wallet becomes root authority; they then choose solo-key or Safe. See the registration-flow note below — it's a commit/reveal with an ERC-20 payment, not a one-shot ETH tx.

C · acme.eth is already owned by an unrelated external party

Don't dead-end. During registration, check label availability and present a few free alternatives — acmehq.eth, acme-dao.eth, acmeclub.eth, getacme.eth — and let them pick one to register as the parent. The org proceeds under the alternative name; nothing about the model changes. (Booth question 5: is there a names-suggestion/availability endpoint, or do you run per-label availability checks yourself?)

v2 registration flow — the concrete steps (from gskril/ens-cli)

Registering on the v2 ETHRegistrar is a commit/reveal with ERC-20 payment, not a single ETH transaction. The reference implementation in ens-cli gives the exact calls:

// 1. (optional, recommended) deploy a per-account resolver via the
//    resolverFactory (0xd2A6…6198); pass its address below. Default = zeroAddress.
// 2. COMMIT
registrar.makeCommitment(label, owner, secret, subregistry, resolver, duration, referrer)
registrar.commit(commitment)              // value: 0
// 3. WAIT ≥ 60 seconds after commit is mined
// 4. APPROVE the registrar to spend the ERC-20 total (paymentToken 0x3DfC…38D9)
// 5. REVEAL / register
registrar.register(label, owner, secret, subregistry, resolver, duration, paymentToken, referrer)  // value: 0
  • Payment is an ERC-20 (paymentToken), not ETH — you must approve() before reveal. This is different from v1 (which sends ETH value). Budget UI for the approval step.
  • subregistry defaults to zero; set it only if the name will itself issue subnames (which yours will → may need a subregistry deployed for the parent).
  • resolver defaults to zero on v2; deploy a per-account resolver first (resolverFactory) so the name can hold profile records.
  • The 60-second commit/reveal wait is real — sequence your demo so you're not staring at a spinner on stage.

Parent control: solo or multisig (owner's choice) + optional Ledger

The registering owner picks the control posture at setup — a forced 2-of-3 would be wrong for a solo club organizer. The authority model is unchanged either way: authority always traces to wallet control of the parent. The "wallet" is just one key or a Safe.

PostureWho controls the parentWhen to use
SoloA single key (ideally a Ledger-backed EOA).One organizer, small club, fast setup.
MultisigA Safe, M-of-N admins (e.g. 2-of-3).Real org, rotating officers, no single point of control.
Where Ledger fits (3rd prize track) Ledger is the device-backed approval layer for high-risk parent operations — transferring the parent, reassigning a subtree (an ENSv2 single-op that wipes a whole branch), or bulk issuance/revocation. It composes either way: a signer on the Safe, or the approval device for a solo owner. This is squarely Ledger's "human-in-the-loop for sensitive actions" example. Note the solo + Ledger case is the strongest security story: with no multisig quorum protecting you, device confirmation is the only thing between a compromised session and a namespace takeover.

Admin onboarding via ENSv2 roles — why ENSv2 earns its place

ENSv2 replaces NameWrapper fuses with a real role-based subsystem called Enhanced Access Control (EAC), built into the Permissioned Registry. Each name has roles (set-resolver, set-subregistry, create-subname…), each role has an admin role that grants/revokes it, roles can be held per-name or contract-wide, and on transfer the roles follow the new owner. That makes your requirements native protocol operations, not database bolt-ons:

  • Onboard a new manager = an existing admin grants the create-subname role on the Permissioned Registry (a solo-key or multisig action depending on posture).
  • "Can issue but can't transfer the parent" = expressible directly in the EAC role grant.
  • No single point of control (multisig posture) = high-risk ops require M-of-N signatures; (solo posture) = high-risk ops require a Ledger device tap.
Confirmed live on Sepolia Contracts are deployed (addresses in §8). EAC is part of the Permissioned Registry at 0xDEDB…B67. Reference build: gskril's ens-cli implements v2 registration against these exact patterns, and the ENSv2 docs ship a "For Contract Developers: build a subname registrar on the Permissioned Registry" tutorial — your two primary references for the onchain branch.
Scope honesty Full role wiring + Safe + a complete permissions console is a lot for a weekend. Build the model correctly but demo at sensible depth: a real 2-of-3 Safe controlling the parent and one role-grant manager-onboarding flow working end-to-end beats a half-working permissions dashboard.

06Issuance: the onchain / offchain toggle

Implement one internal interface with two backends; the manager picks per-org at enrollment. The toggle is not a gimmick — it's the real policy decision from §5, and the abstraction itself reads as "ENS done thoughtfully."

interface SubnameIssuer {
  issue(label, ownerAddress, records, policy): Promise<Subname>
  revoke(fqdn): Promise<void>
}
// OffchainIssuer  → NameStone REST write, CCIP-Read resolution, gasless
                  → works on a v1 OR v2 parent (resolver-layer, see interop note)
// OnchainIssuer   → v2 PermissionedRegistry (0xDEDB…B67) create-subname via EAC role
                  → v1 fallback: NameWrapper.setSubnodeRecord (0x0635…fCE8)
                  ↑ optionally gated by the CRE workflow (see §7)
v1 / v2 interop — NameStone & Durin are safe on a v2 parent The resolver interface did not change between v1 and v2; ENS designed v2 so existing v1 resolvers keep working, with resolution flowing through the Universal Resolver (canonical entry point, upgraded behind the scenes). NameStone and Durin both operate at the resolver layer via CCIP-Read (ERC-3668) — which both versions support — so they're orthogonal to the v1→v2 registry change. Your offchain branch works regardless of whether the parent is v1 or v2: you point the parent's resolver at the gateway, you're not touching the registry. Caveat: resolve through the Universal Resolver and confirm your client library (viem/wagmi/ethers) has current v2 support — library coverage was still catching up in early 2026.

Build order: get the offchain branch fully working end-to-end (login → issue → edit profile) first because it's fastest and gasless, then add the onchain branch as the second implementation behind the same interface. Don't build both half-way in parallel.

07Where Chainlink CRE sits

CRE is the verifiable gate between "email says you're eligible" and "a subname is minted on-chain." It's a trigger → callback → on-chain-write workflow where every step is run across a decentralized oracle network and merged by BFT consensus.

HTTP trigger:
onchain issue requested → CRE callback:
verify eligibility signal
(verified-email attestation / org allowlist / SSO API) → BFT consensus
across DON nodes → EVM write:
authorize issuance
on registrar contract

This satisfies the bounty's hard requirement (Chainlink must cause an on-chain state change — here, the subname issuance), makes the security story real (no single backend or API can forge eligibility — you remove yourself as a trusted party), and stays exactly on-thesis (it's the verification layer for issuance, not a side feature). It also revives your original email-verification idea in its strongest form.

De-risk The bounty states: show a successful simulation via the CRE CLI and their team deploys it to the live network for you during the event. So your qualifying target is a simulating workflow — deployment is Early-Access-gated and you don't need to clear it yourself.
Keep a fallback path CRE has a real learning curve (Go/TS workflows, CLI, trigger/callback model). Keep a plain backend-signed issuance path so your core demo never blocks on CRE being finished. CRE is the prize-qualifying enhancement layered on top, not a dependency of the main thread.

09Sequenced 48-hour build plan

Front-load the one thread that must work flawlessly. A judge needs to see one path end-to-end far more than three paths half-built. Each phase ends in something demoable.

Phase 0 — Foundation

Hrs 0–4
  • Next.js app, wallet/Privy SDK wired, Sepolia RPC + test ETH + paymentToken balance
  • Wire the v2 addresses (§8) + ABIs from ens-cli lib/contracts.ts; resolve via Universal Resolver
  • Register one demo org parent (democlub.eth) on the v2 ETHRegistrar — exercises commit/reveal + ERC-20 approval once, up front
  • Define the SubnameIssuer interface (both backends stubbed)
  • Build the reusable <InfoTip> component early — every control gets one from the start, not retrofitted

Phase 1 — The must-have thread (golden path)

Hrs 4–16

If only this works, you still have a winning ENS demo.

  • Email login via Privy → embedded wallet provisioned
  • Verified email domain → matches pre-enrolled democlub.eth
  • Auto-issue you.democlub.eth via offchain NameStone branch
  • Land in manager → edit text records → build a full ENSv2 profile
  • Resolution works: look up you.democlub.eth and see records

Phase 2 — Breadth: doors + onchain toggle

Hrs 16–28
  • "Connect wallet" + "type .eth name → SIWE" doors into the same session
  • Self-service org enrollment: existing-owner path (connect + prove control of a parent)
  • Onchain issuance branch behind the toggle (ENSv2 role grant; NameWrapper fallback)
  • Register-a-new-name flow for orgs/users without one
  • Name-taken handling: if the desired parent is owned externally, show 3–4 available alternatives to register instead
  • Control-posture toggle at registration: solo key vs Safe multisig

Phase 3 — Prize-qualifying layers

Hrs 28–40
  • Chainlink CRE: workflow that verifies eligibility → writes issuance authorization; get it simulating via CLI (qualifies the prize)
  • Ledger: device-backed approval gate on a high-risk parent op (transfer / subtree reassign / bulk issue) — works for both solo-owner and Safe-signer postures
  • Safe multisig path on democlub.eth + one role-grant manager-onboarding flow (multisig posture demo)

Phase 4 — Polish, demo & submission

Hrs 40–48
  • Record the ≤3-min demo video around the golden path, then show breadth
  • Architecture diagram in README (Arc/CRE bounties explicitly require a diagram)
  • Open-source repo + clear README per ENS qualification rules; no hard-coded values
  • Prep the ENS-booth in-person presentation (Sunday morning)
  • Submit to each track with a one-paragraph "how X is used" note
Cut-line discipline If you fall behind, the priority order is: Phase 1 golden path > CRE simulation > onchain toggle > Ledger approval > Safe/roles > extra doors. Never sacrifice a flawless golden-path demo to add a second half-working integration.
